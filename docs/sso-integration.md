исывающих прое# План интеграции SSO УрФУ

## Обзор

Аутентификация пользователей через единую учётную запись УрФУ.
На этапе MVP используется stub-аутентификация; переход на реальный SSO — по готовности.

## Варианты интеграции

### Вариант 1: OAuth2 / OIDC (рекомендуемый)

```
┌──────────┐    1. Redirect     ┌─────────────┐
│ Frontend │ ──────────────────► │ SSO УрФУ    │
│          │                    │ (OAuth2)    │
│          │ ◄────────────────── │             │
│          │    2. Auth code     └─────────────┘
└────┬─────┘
     │ 3. Auth code
     ▼
┌──────────┐    4. Token req    ┌─────────────┐
│ Backend  │ ──────────────────► │ SSO УрФУ    │
│          │ ◄────────────────── │ Token URL   │
│          │    5. Access token  └─────────────┘
│          │
│          │    6. Userinfo req  ┌─────────────┐
│          │ ──────────────────► │ SSO УрФУ    │
│          │ ◄────────────────── │ Userinfo    │
│          │    7. User data     └─────────────┘
└──────────┘
```

**Плюсы:** пароль пользователя не проходит через наш сервер.
**Минусы:** требуется регистрация OAuth2-клиента в IT-службе УрФУ.

### Вариант 2: LDAP bind

```
┌──────────┐   login/password   ┌──────────┐   LDAP bind    ┌──────────┐
│ Frontend │ ──────────────────► │ Backend  │ ──────────────► │ LDAP     │
│          │                     │          │ ◄────────────── │ УрФУ     │
│          │ ◄────────────────── │          │   bind result   └──────────┘
│          │    JWT token        └──────────┘
└──────────┘
```

**Плюсы:** проще в настройке, не требует регистрации клиента.
**Минусы:** пароль проходит через наш backend (требуется TLS).

## Рекомендация

**OAuth2/OIDC** — если IT-служба УрФУ предоставляет OAuth2 endpoint.
**LDAP** — как fallback, если OAuth2 недоступен.

## Данные от SSO

Ожидаемый формат данных пользователя:

```json
{
  "student_id": "12345678",
  "email": "ivanov@urfu.ru",
  "full_name": "Иванов Иван Иванович",
  "faculty": "РТФ",
  "group": "РИ-123456"
}
```

Маппинг на роли приложения:
- По умолчанию: `студент-участник`
- Капитан команды: назначается внутри приложения
- Организатор: назначается вручную админом

## Stub-аутентификация для dev/test

Пока SSO не подключён, backend использует stub-провайдер:

- Принимает любой email вида `*@urfu.ru` с паролем `test`
- Возвращает моковые данные пользователя
- Интерфейс провайдера идентичен реальному — переключение через env:
  ```
  SSO_ENABLED=false   # stub
  SSO_ENABLED=true    # реальный SSO
  SSO_PROVIDER=oauth2 # или ldap
  ```
